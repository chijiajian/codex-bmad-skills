# Parallelization Plan - Inventory Portal API (Example)

> Generated by `bmad:parallel-plan` / `$bmad-parallel-plan`
> Intent: Create
> Base branch: `main`
> Max parallel workstreams: 3

This is an illustrative Codex BMAD artifact. It shows the shape of a
conflict-free wave plan after stories have reached `ready-for-dev`. The plugin
plans the waves and handoff context only; external development tools create
worktrees, implement stories, verify work, and open pull requests.

## Inputs

| Input | Source |
| --- | --- |
| Story backlog | `bmad-output/stories/` |
| Sprint status | `bmad-output/sprint-status.yaml` |
| Architecture | `bmad-output/architecture.md` |
| Dependency graph | `bmad-output/dependency-graph.json` |
| Wave plan data | `bmad-output/waves.json` |

Shared or cross-cutting scopes:

- `src/types/`
- `src/auth/`
- `src/db/migrations/`
- `src/platform/config.ts`

Stories in the same wave must not share owned file scope or touch the same
cross-cutting scope unless the architecture explicitly makes that safe.

## Wave 1 - Foundation

Run these stories concurrently from `main`:

| Story | Title | Branch | Owned scope |
| --- | --- | --- | --- |
| `1.1` | Project shell and API bootstrap | `story/1.1-api-bootstrap` | `package.json`, `tsconfig.json`, `src/app.ts`, `src/server.ts` |
| `1.2` | Initial inventory schema | `story/1.2-inventory-schema` | `src/db/migrations/001_inventory.sql`, `src/db/schema/inventory.ts` |
| `1.3` | Product catalog model | `story/1.3-product-model` | `src/catalog/product.ts`, `src/catalog/product.repository.ts` |

Why these can run together:

- No story depends on another wave-1 story.
- The database migration story owns only migration slot `001`.
- The catalog model does not touch app bootstrap files.

Merge order:

1. `story/1.1-api-bootstrap`
2. `story/1.2-inventory-schema`
3. `story/1.3-product-model`

Integration checkpoint:

- Confirm the bootstrap imports no catalog implementation detail.
- Confirm migration naming follows the architecture decision record.
- Confirm story-owned files match the declared scopes.

## Wave 2 - Authentication and Stock Read Model

| Story | Title | Branch | Owned scope |
| --- | --- | --- | --- |
| `2.1` | Operator identity model | `story/2.1-operator-identity` | `src/auth/operator.ts`, `src/auth/operator.repository.ts` |
| `3.1` | Stock level read model | `story/3.1-stock-read-model` | `src/stock/stock-level.ts`, `src/stock/stock.repository.ts` |

Why these can run together:

- Both depend on foundation schema from wave 1.
- Auth and stock modules have disjoint owned scopes.
- Neither story changes shared request middleware yet.

Merge order:

1. `story/2.1-operator-identity`
2. `story/3.1-stock-read-model`

Integration checkpoint:

- Confirm both modules use the same repository error conventions.
- Confirm no shared type was introduced outside the declared owned scope.

## Wave 3 - Protected Routes

Run after wave 2 is merged:

| Story | Title | Branch | Owned scope |
| --- | --- | --- | --- |
| `2.2` | Auth middleware | `story/2.2-auth-middleware` | `src/auth/require-operator.ts`, `src/auth/session.ts` |
| `3.2` | Stock level route | `story/3.2-stock-route` | `src/stock/stock.routes.ts`, `src/stock/stock.presenter.ts` |

Potential conflict:

- `3.2` depends on the auth middleware behavior from `2.2`.
- If the route must import `require-operator.ts` directly, move `3.2` to the next
  wave. If the architecture defines a stable middleware contract, these can remain
  in one wave with a merge order constraint.

Selected plan:

1. Merge `story/2.2-auth-middleware`.
2. Rebase `story/3.2-stock-route` on the merged middleware.
3. Merge `story/3.2-stock-route`.
